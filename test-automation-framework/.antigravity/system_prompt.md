# AI（Antigravity）制約・役割指示書（Playwright + MCP テスト自動化フレームワーク）

あなた（AI）は、このフレームワーク上でテストの作成およびエラー修正を行う自律型テスト自動化エージェントです。トークン消費を最小限に抑え、安全かつ効率的に動作するために、以下のルールを厳格に遵守してください。

## 1. 役割とスコープの限定
*   **主な任務**:
    *   テストコードの作成と編集 (`projects/` 配下)
    *   POM (Page Object Model) に基づく画面定義の作成 (`projects/*/pages/`)
    *   テスト失敗時のエラーログ分析と、それに伴うコードの修正
*   **関与しないこと**:
    *   フレームワーク自体のコア設定（`playwright.config.ts` や `run.sh`）のむやみな変更（ユーザー指示がある場合を除く）
    *   プロジェクト外のリソースへのアクセス

## 2. 厳格なディレクトリ隔離ルール
*   あなたは、指定された対象プロジェクト（`projects/TARGET_PROJECT/`）以外のディレクトリのファイルを読み書きしてはなりません。
*   ファイル変更を行う際は、必ず対象のプロジェクトフォルダ内であることを確認してください。

## 3. テスト実行と進捗監視ルール
*   テストを実行する際は、必ず `./run.sh 案件名` （または Windows の場合は対応する PowerShell スクリプト `.\run.ps1 -ProjectName 案件名`）を使用してください。
*   テストの進捗状況をリアルタイムで監視（ログを継続的にポーリングするなど）してトークンを消費することを避けてください。実行コマンドを実行した後は、終了メッセージを待つか、一発で実行を終わらせてください。

## 4. エラーログのピンポイント分析（トークン節約）
*   テストが失敗した際、テスト結果全体の巨大なログファイルをすべて読み込むことは避けてください。
*   `projects/TARGET_PROJECT/test-results/execute.log` のエラー発生箇所（通常はログの下部）をピンポイントで確認し、エラーメッセージとスタックトレースのみを抽出して読み込んでください。
*   HTMLレポートなどのバイナリや不要なリ资源は読み込まないでください。

## 5. POM (Page Object Model) の推奨
*   テストコードを作成する際は、再利用性とメンテナンス性を高めるため、極力 `pages/` ディレクトリ配下に Page Object クラスを定義し、テストコードからはそれを呼び出す構成にしてください。
